# AI 위탁운용 최종 상용화 스펙

- 작성일: 2026-07-22
- 상태: 사용자 검토 대기
- 선행 조건: `docs/superpowers/specs/2026-07-22-ai-fund-stability-v1-design.md` 완료
- 범위: 개인 투자자용 자동투자 서비스에서 통상 제공되는 전략·통제·관제 기능을 AI 위탁운용에 단계적으로 적용

## 1. 목표

최종본은 1차 안정화 기반 위에서 사용자가 선택한 위험 수준과 전략을 자동으로 실행하고, 운영자가 모든 자금·주문·성과·장애를 추적하며, 서비스가 장애 상황에서 보수적으로 멈추는 상용 수준의 AI 위탁운용을 목표로 한다.

이 문서는 투자 수익을 보장하지 않는다. 모든 기능은 손실 가능성을 전제로 하며, 전략 성과와 실거래 결과를 분리해 표시한다.

## 2. 상용 자동투자 사례에서 반영할 기능

### 2.1 로보어드바이저형

일반 로보어드바이저는 투자 목적, 기간, 수입, 위험 성향을 바탕으로 포트폴리오를 구성·관리한다. 본 서비스에는 다음 방식으로 반영한다.

- 위험 성향과 목표 기간에 따른 전략·손실 한도·최대 투자 비중 프로필
- 목표 자산 배분과 정기 리밸런싱
- 성과, 위험, 수수료, 보유 자산을 분리한 사용자 보고서
- 전략 변경 시 즉시 전량 전환이 아닌 예약·점진 적용 정책

### 2.2 거래 봇형

상용 암호자산 봇은 다중 거래소, API 키 비수탁 연결, DCA·그리드, 손절·익절, 트레일링, 부분 익절, 백테스트·모의투자, 외부 신호 연동, 운영 알림을 주로 제공한다. 본 서비스는 이를 다음 우선순위로 반영한다.

- 부분 익절과 본전 손절을 포함한 포지션 종료 정책
- 트레일링 손절·익절
- DCA와 그리드 전략 템플릿
- 백테스트, 워크포워드 검증, PAPER 검증 뒤 CANARY 승격
- 웹훅 신호를 포함한 신호 입력 표준화
- 다중 거래소 자산·주문·성과 통합 보기

## 3. 제품 구조

최종 구조는 다음 네 계층으로 구분한다.

| 계층 | 역할 |
| --- | --- |
| 전략 계층 | AI 신호, 규칙 전략, 외부 웹훅 신호를 표준 신호로 변환 |
| 포트폴리오 계층 | 위험 성향, 목표 배분, 전략별 예산, 리밸런싱 결정 |
| 실행 계층 | 1차 스펙의 어댑터·주문 상태 머신·원장을 사용해 거래소 주문 실행 |
| 운영 계층 | 모니터링, 알림, 대사, 감사, 장애 대응, 릴리스 통제 |

전략은 직접 주문을 제출하지 않는다. 전략은 표준 `TradeIntent`만 만들고, 포트폴리오·리스크·실행 계층이 순서대로 승인해야 주문이 생성된다.

## 4. 전략 기능

### 4.1 공통 종료 정책

- 최초 손절가와 목표가를 주문 생성 시 기록한다.
- 최대 3개의 부분 익절 구간을 지원하며, 각 구간의 매도 비중 합계는 100%를 넘지 않는다.
- 첫 목표가 체결 뒤 남은 수량의 손절 기준을 평균 진입가 이상으로 이동하는 본전 손절을 지원한다.
- 트레일링은 기준 가격, 추적 폭, 활성화 조건을 명시적으로 저장한다.
- 종료 조건이 동시에 충족될 때는 손실 제한, 긴급 중단, 수동 중단, 전략 종료 순으로 우선한다.

### 4.2 DCA와 그리드

- DCA는 최대 진입 횟수, 단계별 예산, 단계 간 최소 시간, 전체 전략 손실 한도를 필수로 둔다.
- 그리드는 가격 범위, 격자 수, 주문당 금액, 범위 이탈 정책, 전체 중단 조건을 필수로 둔다.
- 두 전략 모두 별도 전략 예산을 사용하며, 전체 위탁 배정 자금과 포트폴리오 위험 한도를 넘지 않는다.
- 시장 데이터가 오래되었거나 변동성 제한을 초과하면 신규 격자·추가 매수를 중단한다.

### 4.3 신호 입력

- AI 모델, 규칙 기반 지표, 외부 웹훅, 수동 승인 신호를 공통 `TradeIntent`로 정규화한다.
- 모든 신호에는 생성 시각, 심볼, 방향, 신뢰도 또는 근거, 유효 기간, 출처 식별자가 있어야 한다.
- 동일 출처의 같은 신호는 멱등 키로 한 번만 처리한다.
- 외부 웹훅은 서명 검증, 허용 심볼, 요청 빈도 제한, 재전송 방지 키를 적용한다.

## 5. 포트폴리오와 위험 관리

### 5.1 위험 프로필

보수·중립·공격·사용자 정의 프로필은 신호 신뢰도뿐 아니라 최대 종목 수, 종목별 비중, 거래소별 노출, 일간·주간 손실 한도, 전략별 예산, 허용 레버리지를 함께 정의한다. 레버리지는 거래소 capability와 사용자의 별도 동의가 모두 충족될 때만 지원한다.

### 5.2 회로 차단기

- 일간·주간 손실 한도 초과
- 가격 데이터 지연 또는 급격한 괴리
- 거래소 인증·주문·조회 연속 실패
- 대사 불일치 또는 미확정 주문 누적
- 전략별 최대 연속 손실 횟수 초과

회로 차단기는 신규 진입을 먼저 중단한다. 포지션 강제 청산은 사용자가 사전에 선택한 비상 청산 정책이 있을 때만 실행한다. 그렇지 않으면 포지션은 유지하고 경보와 검토 대기를 남긴다.

### 5.3 성과 계산

- 실현손익, 평가손익, 수수료, 슬리피지, 입출금은 분리한다.
- 전략 성과는 벤치마크와 비교하되, 백테스트·PAPER·LIVE 성과를 절대 혼합하지 않는다.
- 수익률은 기간 수익률, 최대 낙폭, 변동성, 승률, 평균 손익비, 회전율을 함께 제공한다.

## 6. 운영 콘솔과 사용자 경험

### 6.1 운용 모드와 승격

각 전략과 거래소 조합은 `BACKTEST`, `PAPER`, `CANARY`, `LIVE`, `PAUSED`, `HALTED` 상태를 가진다. LIVE 승격은 다음 조건을 모두 만족해야 한다.

- 데이터 품질과 어댑터 계약 테스트 통과
- PAPER 기간의 사전 정의된 최소 표본 수 충족
- CANARY 손실 한도와 주문 성공률 기준 충족
- 운영자 승인 기록 존재

### 6.2 관리자 관제

- 거래소별 연결 상태, API 인증 상태, 마지막 시세 시각, 워커 심박, 대사 결과
- 설정·전략·주문·체결·원장 간 드릴다운 조회
- `NEEDS_REVIEW` 주문을 원인과 함께 분류하고, 재조회·취소 요청·수동 조정의 감사 로그 보관
- 긴급 중단, 거래소 단위 중단, 전략 단위 중단, 사용자 단위 중단
- 일간 요약과 장애 이력

### 6.3 사용자 화면

- 위탁 배정 자금, 사용 가능 금액, AI 귀속 포지션, 진행 중 주문, 실현·평가손익을 구분해 표시한다.
- 전략 시작 전 위험 프로필, 거래소, 운용 모드, 최대 손실 한도, 종료 정책을 확인한다.
- 위험 경고와 중단 상태는 성과 카드보다 우선해 표시한다.
- 자동 매매가 멈춘 이유와 재개 조건을 구체적으로 보여 준다.

## 7. 데이터·감사·보안 원칙

- API 키는 사용자별 암호화 저장, 최소 권한, 출금 권한 미사용을 원칙으로 한다.
- 주문 의도, 위험 판정, 거래소 요청 결과, 체결, 원장 조정, 운영자 조치는 시간순 감사 기록으로 남긴다.
- 민감정보와 거래소 원본 응답은 로그 마스킹·접근 제어를 적용한다.
- 관리자 권한 검증은 최종 상용화 전에 Flask API·Supabase RLS·운영 도구 경로 전체에 일관되게 적용해야 한다.
- 전략 설정 변경은 적용 시각과 변경 전후 값을 기록하고, 실행 중 주문에 소급 적용하지 않는다.

## 8. 단계별 로드맵

| 단계 | 산출물 | 선행 조건 |
| --- | --- | --- |
| 1차 안정화 | 거래소 어댑터, 위탁 원장, 주문 상태 머신, 대사, PAPER/CANARY/LIVE | 없음 |
| 2차 전략 종료 | 부분 익절, 본전 손절, 트레일링, 전략별 예산 | 1차 완료 |
| 3차 전략 확장 | DCA, 그리드, 표준 신호·웹훅, 백테스트 | 2차의 공통 주문·원장 검증 |
| 4차 포트폴리오 | 위험 성향, 배분, 리밸런싱, 성과 보고 | 3차 전략 데이터 축적 |
| 5차 상용 관제 | 승격 절차, 알림, 운영 콘솔, 전면 권한 검증 | 모든 LIVE 경로 검증 |

각 단계는 독립적으로 PAPER에서 검증하고, 측정 가능한 기준을 충족한 조합만 다음 모드로 승격한다. 최종본의 기능을 1차에 함께 넣지 않는다.

## 9. 최종 완료 정의

최종 상용화는 다중 거래소에서 AI 위탁 자금과 포지션을 정확히 추적하고, 전략별 위험·성과·주문 이력을 분리하며, 장애·불일치·손실 한도 초과 때 신규 진입을 보수적으로 차단하는 상태다. 어떤 자동 전략도 공통 리스크, 주문 상태 머신, 원장, 감사 기록을 우회해 거래소 주문을 직접 제출할 수 없어야 한다.
